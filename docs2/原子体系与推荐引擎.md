# 原子体系与推荐引擎

> 这是整个产品的操作系统。卡片、画像、推荐、未来的 Agent 全部建立在这套原子之上。
> 设计原则：最小充分集，每个原子都必须能被用户行为观测到。

---

## 一、原子体系

### 设计约束

1. **可观测**：每个原子都必须能从用户行为中推断，不能依赖用户自述
2. **正交**：原子之间尽量不重叠，避免一个行为同时强更新多个相关维度
3. **可组合**：有限原子的组合能覆盖所有合理卡片
4. **可量化**：每个原子有连续值，不是二元标签

### A. 体验原子（描述一张卡片"是什么"）

两层：**动作层** + **情境层**。

#### 动作原子 — 7 个

砍掉了上版的 `collect` 和 `break`。`collect` 本质是 `observe` 的子类，`break` 本质是对其他原子的"反向使用"，不是独立动作。

| 原子 | 语义 | 可观测行为 |
|------|------|-----------|
| `explore` | 去没去过的地方 | 完成含位移的卡片 |
| `taste` | 尝试没试过的东西 | 完成"第一次做 X"类卡片 |
| `create` | 产出一个东西 | 完成有产出物的卡片 |
| `connect` | 和人发生连接 | 完成含社交的卡片 |
| `move` | 身体运动 | 完成含体力消耗的卡片 |
| `observe` | 感受、观察、收集 | 完成感知类卡片 |
| `reflect` | 向内审视 | 完成内省类卡片 |

每张卡片标注 1-2 个动作原子，主原子权重 1.0，次原子权重 0.5。

#### 情境原子 — 5 个维度

砍掉了上版的"计划性"维度。原因：计划性是用户侧状态，不是卡片固有属性——同一张卡可以即兴做也可以计划做。

| 维度 | 值域 | 编码 |
|------|------|------|
| `social` | `solo` / `pair` / `group` / `stranger` | 枚举，可多选 |
| `space` | `home` / `nearby` / `city` / `nature` | 枚举，单选 |
| `time` | 5 / 30 / 120 / 480（分钟） | 连续值 |
| `cost` | 0 / 50 / 200 / 500（元） | 连续值 |
| `intensity` | 0.0 - 1.0 | 连续值，心理门槛 |

### B. 画像原子（描述一个用户"是谁"）

#### 偏好向量 P（你喜欢什么）

7 维，对应 7 个动作原子。每维是 -1.0 到 1.0 的连续值。

```
P = [explore, taste, create, connect, move, observe, reflect]

初始值：[0, 0, 0, 0, 0, 0, 0]（无偏好）
```

#### 情境偏好 C（你习惯在什么条件下行动）

```
C = {
  social: { solo: 0.5, pair: 0.3, group: 0.1, stranger: 0.1 },
  space:  { home: 0.3, nearby: 0.4, city: 0.2, nature: 0.1 },
  time:   { mean: 60, std: 40 },     // 偏好的时间投入分布
  cost:   { mean: 30, std: 50 },     // 偏好的花费分布
  intensity: { mean: 0.3, std: 0.2 } // 偏好的强度分布
}
```

初始值均匀分布，随交互更新。

#### 状态向量 S（你现在处于什么内在状态）

6 维。这就是星图中"星座"的底层数据。

| 维度 | 度量什么 | 怎么涨 |
|------|---------|--------|
| `vitality` | 身体活力 | 完成含 `move` 的卡片 |
| `curiosity` | 对未知的开放度 | 完成含 `explore` / `taste` 的卡片 |
| `bond` | 和他人的连接 | 完成含 `connect` 的卡片 |
| `ground` | 和物理世界的关系 | 完成 `space ≠ home` 的卡片 |
| `agency` | 行动力 | 完成任何卡片（做了就涨） |
| `depth` | 内在认知深度 | 完成含 `reflect` / `observe` 的卡片 |

每个维度 0-100，半衰期 30 天。

**S 和 P 的区别**：P 是"你喜欢什么"（指导推荐），S 是"你最近活成了什么样"（呈现给用户看的画像）。P 是输入，S 是输出。

## 二、信号采集与权重更新

### 用户行为 → 信号

| 行为 | P 更新（偏好） | C 更新（情境偏好） | S 更新（状态） |
|------|--------------|------------------|--------------|
| 右滑（想做） | +0.15 对应原子 | 无 | 无 |
| 左滑-不感兴趣 | -0.10 对应原子 | 无 | 无 |
| 左滑-现在不合适 | 无 | 无 | 无 |
| 停留 >3s 后左滑 | 无（犹豫≠不喜欢） | 无 | 无 |
| 展开预演 | +0.05 对应原子 | 无 | 无 |
| **打卡完成** | **+0.5 对应原子** | **更新情境分布** | **+对应维度** |
| 打卡情绪正向 | 额外 +0.2 | 无 | 额外 +0.15 |
| 打卡情绪"平淡" | 无额外 | 无 | 无额外 |
| 打卡情绪"疲惫" | 无额外 | intensity偏好略下调 | 正常更新 |

**核心原则：行动的权重远大于意愿。** 右滑是 0.15，打卡是 0.5。做了 > 想做 > 看了。

### 衰减机制

```
每天衰减：
P[i] *= 0.98   // 偏好 30 天半衰期
S[i] *= 0.977  // 状态 30 天半衰期

含义：如果你 30 天没做 explore 类的事，
explore 偏好回到接近 0（中性），
curiosity 状态也回到接近 0。

你的画像反映的永远是"最近的你"，不是"曾经的你"。
```

## 三、推荐引擎

### 流水线

```
                    全量卡片池
                        │
                        ▼
              ┌─── ① 硬过滤 ───┐
              │                │
              │  已完成的       │
              │  已拒绝的       │
              │  3天内展示过的   │
              │  情境不可能的    │
              │                │
              └───── 剩余 ─────┘
                        │
                        ▼
              ┌─── ② 情境匹配 ──┐
              │                 │
              │  时刻适合？      │
              │  天气允许？      │
              │  位置可行？      │
              │  节气/季节命中？  │
              │                 │
              │  → 每张卡得到    │
              │    context_score │
              └────────────────┘
                        │
                        ▼
              ┌─── ③ 画像匹配 ──┐
              │                 │
              │  偏好向量点积    │
              │  情境偏好距离    │
              │                 │
              │  → 每张卡得到    │
              │    profile_score │
              └────────────────┘
                        │
                        ▼
              ┌─── ④ 策略层 ───┐
              │                │
              │  强度渐进       │
              │  多样性保障     │
              │  野卡注入       │
              │  每日精选       │
              │                │
              │  → 最终排序     │
              └────────────────┘
                        │
                        ▼
                   卡片栈（~15张）
```

### ② 情境匹配 — 打分细则

```
context_score(card, now) =
    time_fit(card, now.hour, now.weekday) × 0.3
  + weather_fit(card, now.weather)        × 0.25
  + space_fit(card, now.location)         × 0.25
  + season_fit(card, now.solar_term)      × 0.2

其中：

time_fit：
  card 的 timeOfDay 标签与当前时段匹配 → 1.0
  不匹配但不冲突 → 0.5
  冲突（深夜推户外） → 0.0

weather_fit：
  card.space = nature/city + 暴雨 → 0.0
  card.space = home + 晴天周末 → 0.3（不是不行，但浪费了好天气）
  天气反差命中（雨后初晴 + 出门类） → 1.5（加成）

space_fit：
  用户在家 + card.space = home → 1.0
  用户在家 + card.space = nearby → 0.7（要出门但不远）
  用户在家 + card.space = city → 0.3
  用户在外 + card.space = home → 0.1
  用户在外 + card.space 匹配当前位置类型 → 1.2（已经在外面了，加成）

season_fit：
  节气卡命中当前节气窗口 → 2.0（稀有加成）
  季节卡命中当前季节 → 1.0
  反季 → 0.0
```

### ③ 画像匹配 — 打分细则

```
profile_score(card, user) =
    action_fit(card.actions, user.P)     × 0.5
  + context_fit(card.context, user.C)    × 0.3
  + growth_fit(card, user.S)             × 0.2

其中：

action_fit = dot(card.action_vector, user.P)
  card.action_vector：主原子=1.0，次原子=0.5，其余=0
  user.P：当前偏好向量
  结果：越匹配偏好，分越高

context_fit = 1 - distance(card.context, user.C)
  衡量卡片的情境条件和用户习惯的接近程度
  用户习惯独处 + 卡片是 solo → 高分
  用户习惯低花费 + 卡片是高花费 → 低分

growth_fit：这是关键的差异化
  看用户 S 向量中哪个维度最低 → 那个维度对应的卡片获得加分

  原理：不只推你喜欢的，也轻轻推你缺少的。
  如果你 bond 很低（很少做社交类的事），
  社交类卡片会获得 growth 加分——但不会压过偏好分，
  只是让它"偶尔出现"而非"永远消失"。
```

### ④ 策略层

#### 强度渐进

```
用户累计打卡次数 → 决定可推的最高 intensity

0-2 次：intensity ≤ 0.3（只推低门槛）
3-7 次：intensity ≤ 0.6
8+ 次：不限制

原理：新用户还没建立信任。推高强度的卡会吓跑人。
等用户做了几件事、感受到价值后，再慢慢提高门槛。
```

#### 多样性保障

```
最终卡片栈中：
- 同一个主动作原子最多连续 2 张
- 至少包含 1 张 social 类（即使用户偏好 solo）
- 至少包含 1 张用户画像低维度对应的卡片
- 相邻卡片的 intensity 差异 > 0.2（避免连续高压或连续低压）
```

#### 野卡

```
每 5 张卡片中插入 1 张"野卡"：
- 从画像匹配分最低的 20% 中随机选一张
- 但 context_score 必须 > 0.5（情境得合适）
- intensity ≤ 用户当前可接受上限

野卡的意义：
1. 防信息茧房——你可能不知道自己喜欢什么
2. 采集探索信号——用户对野卡的反应是最有信息量的数据
3. 制造惊喜——"这个我没想到，但好像可以试试"
```

#### 每日精选

```
每天第一张卡的选择逻辑独立于正常排序：

daily_pick_score =
    context_score × 0.4        // 此刻合不合适最重要
  + profile_score × 0.3        // 要符合偏好
  + novelty_score × 0.3        // 没见过的优先

novelty_score：
  从未展示过 → 1.0
  展示过但未右滑 → 0.3
  展示过且右滑但未完成 → 0.1
  展示过且完成 → 0.0

节气窗口内的节气卡自动成为当天精选（覆盖常规逻辑）。
```

## 四、冷启动策略

### 无数据时（第 1-5 张卡）

P = 零向量，画像匹配无意义。此时完全依赖情境匹配 + 人工编排：

```
第 1 张：普世微粒卡，intensity ≤ 0.2，任何人都能共鸣
第 2-5 张：覆盖不同动作原子的低强度卡，用于采集初始信号

前 5 张的动作原子分布：
  explore × 1, connect × 1, observe × 1, reflect × 1, taste × 1
  （覆盖 5/7 个原子，create 和 move 延后，因为门槛相对高）
```

### 快速收敛（第 5 张之后）

5 次右滑/左滑后，P 向量已经不是零了。开始有方向。

此时触发第一次画像确认：

> 「你好像更喜欢 __（P 最高维）类的体验。」

用户看到这句话时的反应本身也是信号——如果他继续右滑类似的卡，确认了；如果开始左滑，说明初始信号有噪音，P 会自然修正。

### 收敛速度

```
模拟计算：
- 10 次右滑/左滑后，P 的方向基本稳定
- 3 次打卡后，P 的绝对值开始有区分度
- 5 次打卡后，推荐开始明显个性化

用户感知：
- 前 5 张："还行，有意思"
- 第 5-10 张："嗯这几张都挺合我的"
- 第 10 张之后："它怎么知道的？"
```

## 五、为 Agent 化预留的数据基石

当前体系的每一个数据都是未来 Agent 的燃料：

| 当前数据 | 未来 Agent 用法 |
|---------|----------------|
| P 偏好向量的时间序列 | 「你最近三个月越来越喜欢独处了。发生什么了吗？」 |
| S 状态向量的变化曲线 | 「你的 bond 维度上个月降了很多。要不要试试社交类的体验？」 |
| 打卡的焦点回答 | 「你去过的 5 家餐厅都是暖色灯光——你可能喜欢温暖感的空间」 |
| 情绪选择的分布 | 「你做完社交类的事大多选'疲惫'而非'开心'——你可能是那种喜欢朋友但社交耗电快的人」 |
| 野卡的反应模式 | 「你对野卡的接受率异常高——你比大多数人更愿意尝试新事物」 |
| 情境偏好 C 的漂移 | 「去年你几乎不做户外活动，今年你做了 12 次——你变了」 |
| 暗星的生命周期 | 「你有 7 颗暗星超过 21 天了。它们大多是 connect 类的——你想做但一直没做社交类的事。什么在阻碍你？」 |

**关键：这些洞察不需要 LLM 的推理能力。需要的是结构化的、干净的、有时间维度的原子级数据。** 当前体系的设计就是在确保数据质量。

### Agent 画像协议（预留结构）

```typescript
interface UserProfile {
  // 静态
  id: string
  created_at: timestamp

  // 动态画像
  preference: number[]        // P 向量，7 维
  context_pref: ContextPref   // C 情境偏好
  state: number[]             // S 向量，6 维

  // 行为日志（原子级）
  events: Event[]             // 每次交互的原子记录

  // 洞察缓存
  insights: Insight[]         // 已触发的洞察
}

interface Event {
  card_id: string
  action: 'swipe_right' | 'swipe_left_dislike' | 'swipe_left_timing' | 'expand' | 'checkin'
  timestamp: timestamp
  context_snapshot: {         // 行为发生时的情境快照
    hour: number
    weekday: number
    weather: string
    location_type: string
    solar_term: string
  }
  checkin_data?: {
    emotion: string
    focus_answer: string      // 焦点追问的回答
    note: string
  }
}
```

每一条 Event 都是原子级的、带情境快照的、可回溯的。这是未来一切智能的原材料。
