# 卡片如何感知"当下"

> 推荐的终极目标不是"推对了"，而是让用户觉得"它好像知道我现在的状态"。

---

## 一、重新理解"当下"

"当下"不只是时间。它是多个维度叠加的一个瞬间状态：

| 维度 | 可获取？ | 怎么拿 |
|------|---------|--------|
| 时刻 | ✅ 免费 | 系统时间 |
| 星期几 | ✅ 免费 | 系统时间 |
| 天气 | ✅ 免费 API | 公开天气接口 |
| 季节/节气 | ✅ 免费 | 本地日历计算 |
| 粗略位置 | ✅ 授权后 | 系统定位（城市级即可） |
| 日出日落 | ✅ 免费 | 经纬度 + 日期计算 |
| 用户刚醒/快睡 | ⚠️ 可推断 | 打开时间 + 历史模式 |
| 用户在家/在外 | ⚠️ 可推断 | 位置是否在常驻区域 |
| 用户的能量水平 | ⚠️ 可推断 | 时间 + 行为模式 |
| 用户的情绪状态 | ❌ 不能 | 不猜。猜错比不猜更糟 |

这些维度叠加起来，描述了一个远比"用户画像"更即时、更具体的状态。

---

## 二、"情境感知"比"画像匹配"更有穿透力

### 为什么

画像说的是"你是什么样的人"。
情境说的是"你现在是什么状态"。

一个喜欢户外的人在雨夜不想出门。
一个内向的人在朋友聚会后可能突然想一个人走走。

**画像是慢变量，情境是快变量。卡片推荐应该先过情境、再过画像。**

### 顺序

```
全量卡片
  │
  ▼
① 情境过滤（当下的环境允许什么）
  │
  ▼
② 画像排序（在允许的范围内，你更可能喜欢什么）
  │
  ▼
③ 多样性 + 野卡
  │
  ▼
呈现
```

情境在前，画像在后。因为再精准的画像匹配，推了一张"去公园坐半小时"给暴雨中的用户，就是失败。

## 三、可用情境信号的深度挖掘

### 3.1 时刻 — 不只是"早中晚"

粗暴的做法：早上推晨跑，中午推探店，晚上推安静的事。

深入的做法：**理解时刻背后的心理状态。**

| 时段 | 不只是"几点" | 更是这种状态 | 适合的卡片 |
|------|------------|------------|-----------|
| 早 7-9 点（工作日） | 准备上班 | 被日程推着走，没有决策余力 | 微粒卡，不需要额外安排的：「今天通勤路上摘下耳机，听听周围的声音」 |
| 上午 10-11 点（周末） | 刚醒不久，一天很长 | 有整块时间的富足感 | 中等强度卡：「去一个你一直想去但没去的地方」 |
| 下午 2-4 点（任何一天） | 午后低能量 | 不想动，注意力涣散 | 极低门槛卡：「把桌面清理一下，只留三样东西」 |
| 傍晚 5-7 点（工作日） | 下班路上/刚到家 | 从工作状态切换出来的转折点 | 路径卡：「换一条路回家」「下班路上买一样今天想吃的东西」 |
| 晚 8-10 点 | 今天快结束了 | 轻微的"今天又没做什么"的空虚感 | 即刻可做的微粒卡 + 安抚调性 |
| 晚 11 点后 | 该睡了但没睡 | 夜晚的清醒：更感性、更诚实 | 内省卡：「写三件今天发生的事，不用好事」 |
| 凌晨 1-5 点 | 深夜/失眠 | 不推卡片。推了也不会做。此刻唯一对的事是 → 「现在什么都不用做。」 |

**关键洞察：同一个用户，在不同时段就是不同的人。** 画像是恒定的，但下午 3 点的你和晚上 11 点的你，需求完全不同。

### 3.2 星期几 — 隐含的自由度

| 日子 | 隐含的心理 | 卡片策略 |
|------|-----------|---------|
| 周一 | 能量最低，抗拒额外负担 | 只推最低门槛的卡 |
| 周二-周四 | 工作惯性中，没有"今天特别"的感觉 | 通勤类/午间类微粒卡，不需要额外时间 |
| 周五晚 | "终于到周末了"的释放感 | 门槛可以高一档，行动冲动强 |
| 周六 | 唯一真正有大块空闲的日子 | 中等/大型卡片的最佳时机 |
| 周日下午 | "明天又要上班了"的焦虑开始渗入 | 安静/治愈类，不推挑战类 |

### 3.3 天气 — 最被低估的情境信号

天气影响心理状态，这不是迷信，是研究反复验证的。

| 天气 | 心理影响 | 卡片策略 |
|------|---------|---------|
| 晴天 | 行动力 +30%，户外意愿高 | 这是推户外卡的最佳时机 |
| 阴天 | 情绪平，没有明确倾向 | 正常推荐 |
| 小雨 | 微微的忧郁 + 舒适的被包裹感 | 室内体验 + 安静调性，是内省卡的最佳天气 |
| 大雨/暴雨 | 被困住的感觉 | 纯室内卡 + 挑战类室内卡（「用左手吃完今天的一顿饭」） |
| 雪 | 稀有感，"今天不一样" | 和雪直接相关的卡是最佳体验：「出去踩雪听声音」 |
| 极端高温 | 不想出门 | 室内，强制过滤所有户外卡 |
| 第一个降温日 | 季节转换的感知力最强的一天 | 季节相关卡：「穿上这个秋天的第一件外套出门走走」 |

**高级用法：天气的"反差"比天气本身更有价值。**

连续晴天后第一天下雨 → 「找一家咖啡馆听听雨」（雨天变成了一种体验，而非障碍）。
连续阴雨后第一天放晴 → 「出门。不用有目的。出门就对了。」（放晴本身就是做事的理由）。

这需要记录前 3-5 天的天气才能判断。技术上很简单，但产品感知上极强。

### 3.4 季节与节气 — 被互联网遗忘的时间刻度

大多数 App 不关心节气。但节气是**中国人身体里最根深蒂固的时间感知**。

你不需要迷信，但——"清明"这两个字本身就带着情绪。"立秋"这两个字自带一种微微的凉意。

| 节气/季节窗口 | 能触发的体验 |
|-------------|------------|
| 立春前后 | 「找到你附近第一朵开的花」 |
| 清明前后 | 「给一个你想念的人写一段话，不用发出去」 |
| 夏至 | 「在最长的白天里，做一件你一直拖着没做的事」 |
| 处暑 | 「今晚在窗边坐一会儿，感受夏天快结束了」 |
| 秋分 | 「换一条路去上班/学校——秋天的树不一样了」 |
| 冬至 | 「今天是最短的白天。早点收工，回家做一顿饭。」 |
| 大寒 | 「最冷的时候，给自己买一样取暖的东西，不用实用。」 |

节气卡是**稀有卡**。一年 24 个，每个窗口只有几天。稀有意味着有仪式感。

### 3.5 位置 — 不需要精准，粗略就够

不追踪用户行踪。只需要知道一件事：**用户在"家附近"还是"外面"。**

获取方式：首次授权时记住一个"家"的粗略位置（区级精度就够）。之后判断当前是否在这个范围内。

| 状态 | 推什么 |
|------|--------|
| 在家附近 | 「去你家 500 米内你没进过的那家店」「在阳台站五分钟」 |
| 在外面（非家附近） | 「你现在在一个不熟悉的地方——拐进最近的巷子看看」 |
| 在外面 + 周末 | 「你已经在外面了，不如走远一点」 |

在外面时推的卡应该比在家时更行动导向——**因为用户已经克服了"出门"这个最大障碍**。此时的行动力最高，是推探索类卡片的最佳时机。

### 3.6 日出日落 — 纯计算，零成本，极强仪式感

有经纬度就能算出每天的日出和日落时间。

| 时机 | 可以做的事 |
|------|-----------|
| 日出前 30 分钟（如果用户此时打开 App） | 「太阳还有 30 分钟升起来。你要是能看到今天的日出——那就是今天的全部。」 |
| 日落前 30 分钟 | 「太阳还有半小时落下。如果你能找到一个看得到天的地方——」 |
| 日落后 10 分钟 | 「你错过了今天的日落。没关系，明天 18:42。」 |

日落时间精确到分钟这件事本身就有力量。它不是 App 在催你看日落，是 App 在告诉你一个你不知道的事实：**今天太阳 18:42 落下。**

### 3.7 用户行为节奏 — 通过使用模式推断

不问用户"你几点起床"。观察：

- 用户连续 3 天都在 8 点打开 App → 大概率是通勤时间
- 用户周末都在 10-11 点首次打开 → 周末晚起
- 用户经常在 23 点后打开 → 夜猫子

这些模式不需要告诉用户"我知道了"。但推荐引擎默默使用它们：

- 通勤时推通勤可做的卡
- 夜猫子不在 23 点推"早睡"（那是说教），推"既然醒着"的卡

## 四、情境信号的叠加 — "恰好"的魔法

单一信号有用但不惊艳。信号叠加才能产生"它怎么知道"的魔法。

### 示例组合

**周五晚 + 晴天 + 在家**：
> 「天还亮着，出门走走。今天天气刚好。」
> — 系统知道你在家 + 天还没黑 + 是周末前夜 + 天气好 → 四个信号叠加

**周日下午 + 阴天 + 在家**：
> 「把这周穿过的衣服整理一下。不是家务，是跟这一周告个别。」
> — 系统知道周日焦虑 + 天气压抑 + 在家 → 推一个把日常行为赋予意义的微粒卡

**连续 5 天没打开 App + 再次打开**：
> 不推新卡片。首屏显示：
> 「好久不见。」
> 然后正常进入。

**第一次降温 + 工作日傍晚**：
> 「今天冷了一截。下班路上买一杯热的。什么都行。」

**用户 7 天内做了 3 件事（高活跃） + 周六**：
> 可以推稍高强度的卡了：「今天有整块时间吧。去一个你一直说'有空去'但从没去的地方。」

### 核心原理

**用户不知道我们用了这些信号。他只是觉得：这个 App 推给我的东西，怎么每次都挺合适的。**

这就是 Spotify 让人上瘾的原因——不是算法有多聪明，是 Discover Weekly 里的歌"恰好"符合你此刻的心情。你不知道为什么，但你信了。

## 五、"感知当下"的终极价值

到目前为止我们都在说"情境感知让推荐更准"。但如果只停在这里，就只是一个更好的推荐引擎。

**更深的价值是：App 帮用户感知到他自己没注意到的"当下"。**

大多数人对当下是麻木的。周五和周二没区别，晴天和阴天没区别，春天和秋天没区别。不是因为他们不在乎，是因为没有人提醒他们"今天和昨天不一样"。

当 App 在清明前推一张「给想念的人写一段话」，用户的反应可能是：

> 「哦，清明快到了。是啊，我确实很久没想起 TA 了。」

App 做的事不只是"推了一张合适的卡"。它做的是**把用户从日常的麻木中轻轻拽出来，让他重新感知到时间在流动、季节在变化、他活在一个具体的"此刻"而非模糊的"每天"**。

这和产品的终极理念完全吻合——"帮助使用者认识自己"。认识自己的前提是：你得先"在场"。而大多数人大多数时候是不在场的。

**App 不是一个事件推荐器。它是一个帮你"在场"的工具。**

## 六、技术实现成本

| 信号 | 实现方式 | 是否需要后端 | Phase |
|------|---------|-------------|-------|
| 时刻/星期 | `new Date()` | 否 | 0 |
| 节气 | 本地农历库（几 KB） | 否 | 0 |
| 日出日落 | 经纬度 + 公式计算 | 否 | 0 |
| 天气 | 免费 API（和风天气等） | 需联网，不需要自建后端 | 0 |
| 天气反差 | 本地缓存近 5 天天气 | 否 | 0 |
| 粗略位置 | 系统定位 API | 否 | 0 |
| 用户行为节奏 | 本地记录打开时间 | 否 | 1 |
| 连续未打开天数 | 本地时间戳对比 | 否 | 0 |

**全部 Phase 0 可实现。全部不需要后端。** 这就是情境感知的 ROI——几乎零成本，但用户感知极强。
