# Vibe 产品重新定义

> 从零开始重新梳理。不是迭代旧文档，是重新想清楚这件事。
> 日期：2026-03-24

---

## 一、这个产品到底在做什么

一句话：**帮你发现生活里你不知道自己会喜欢的事，推你去做，然后通过你真正做了什么来认识你。**

不是攻略平台（小红书已经做了），不是日程管理（滴答清单已经做了），不是性格测试（MBTI 已经做了）。

它做的是一件没人做的事：**用真实行动，而不是问卷、不是浏览记录、不是消费数据，来构建一个人的画像。**

你看了什么只能说明你感兴趣什么。你买了什么只能说明你消费什么。但你**真正去做了什么**——周末选择一个人爬山而不是约朋友喝酒、选择去陌生街区而不是待在家里——这些选择才揭示你是谁。

## 二、原子体系：一切的底层

### 为什么要原子化

原子化不是技术偏好，是产品能运转的根基。原因有三：

1. **卡片可以被组合生成**，而非纯人工编写——有限的原子组合出无限的卡片
2. **用户画像可以被交互反推**——每次选择就是在原子空间里投一票
3. **推荐可以参数化**——不是模糊的"你可能喜欢"，而是精确的原子权重匹配

### 原子体系 v2（重新设计）

旧文档里的原子分了触发/行动/情境/结果四类，逻辑清晰但过于复杂。重新设计的原则：**原子数量要少到可以在脑子里装下，但组合要足够覆盖所有卡片。**

#### A. 行为原子（做什么）— 9 个

这是最核心的一组，直接描述"这件事的本质动作是什么"。

| 原子 | 含义 | 例子 |
|------|------|------|
| `explore` | 去一个没去过的地方 | 陌生街区、终点站、新餐厅 |
| `taste` | 尝试一个没试过的东西 | 新口味、新类型电影、新运动 |
| `create` | 产出一个东西 | 做菜、写信、拍照、画画 |
| `connect` | 和人产生连接 | 打电话、约饭、和陌生人聊天 |
| `move` | 让身体动起来 | 走路、骑车、爬山、游泳 |
| `observe` | 安静地感受 | 发呆、看日落、听完一张专辑 |
| `reflect` | 向内看自己 | 写日记、回忆、复盘 |
| `break` | 打破一个惯性 | 不带手机、不刷社交媒体、换路线 |
| `collect` | 收集或记录什么 | 捡石头、拍照、买一本书 |

#### B. 情境原子（怎么做）— 6 个维度

| 维度 | 值 | 说明 |
|------|------|------|
| 社交性 | `solo` / `pair` / `group` / `stranger` | 独处 / 固定同伴 / 小群体 / 陌生人 |
| 空间 | `home` / `nearby` / `city` / `nature` / `travel` | 在家 / 附近 / 城市 / 自然 / 跨城 |
| 时间 | `5min` / `30min` / `2h` / `halfday` / `fullday` | 最小时间投入 |
| 花费 | `free` / `low` / `medium` / `high` | 0 / <50 / 50-200 / 200+ |
| 计划性 | `impulse` / `sameday` / `planned` | 即兴 / 当天准备 / 提前规划 |
| 强度 | `low` / `medium` / `high` | 心理门槛 |

#### C. 画像原子（影响什么）— 6 个维度

这就是六维画像，但重新定义其含义：

| 维度 | 真正在度量什么 |
|------|--------------|
| `vitality` | 身体活力——你和自己身体的关系 |
| `curiosity` | 好奇心——你对未知事物的开放度 |
| `bond` | 连接——你和他人关系的活跃度 |
| `ground` | 扎根——你和物理世界/所在城市的关系 |
| `agency` | 行动力——你把想法变成行动的能力 |
| `depth` | 深度——你向内审视和自我认知的程度 |

**为什么改掉了"财务"和"事业"维度？**

旧版六维里的"财务"和"事业"有两个问题：
1. 它们无法从产品内行为准确推断——你花了 200 块吃饭不代表财务状况好或坏
2. 它们偏向"社会评价"而非"自我认知"——这个产品的终极目标是帮人认识自己，不是评估社会地位

新六维全部指向**内在状态**，并且每一个都可以通过产品内行为可靠推断。

### 原子组合示例

一张卡片 = 行为原子 + 情境原子的组合：

```
「去一家从没去过的餐厅，不看菜单」
= explore + taste
+ solo/pair, city, 2h, low, impulse, medium

画像影响: curiosity↑↑, ground↑
```

```
「今晚上床不带手机」
= break + reflect
+ solo, home, 5min, free, impulse, low

画像影响: vitality↑, depth↑
```

```
「给一个很久没联系的人发消息」
= connect
+ solo, home, 5min, free, impulse, medium

画像影响: bond↑↑
```

原子组合的强大之处：**可以枚举所有合理组合，发现卡片库的覆盖空白，也可以用 AI 基于原子组合批量生成候选卡片。**

## 三、用户画像构建——交互即采集

### 冷启动：用原子交叉定位

传统做法是让用户选兴趣标签。我们不做这个。

我们做的是：**用几个精心设计的"微选择"来快速锁定用户在原子空间里的初始位置。**

#### Taste Seeding（入口体验，约 60 秒）

不是问卷，不是标签选择。是一组快速的二选一 / 场景选择：

```
第 1 题：场景触发
「周六下午，没有安排。你更想——」
  A. 出门走走，去没去过的地方
  B. 待在家里，找点安静的事做

→ 采集: explore vs reflect, outdoor vs indoor
```

```
第 2 题：社交偏好
「做一件新事的时候，你更倾向——」
  A. 一个人去
  B. 找个人一起

→ 采集: solo vs social
```

```
第 3 题：行动门槛
「看到一个有意思的事情，你通常——」
  A. 想做就做
  B. 想想再说

→ 采集: agency 初始值, impulse vs planned 偏好
```

```
第 4 题：感官偏好
「以下哪个更吸引你——」
  A.「在一家陌生的小店里，点一道你不认识的菜」
  B.「找一个安静的地方，把最近的事情想一想」

→ 采集: taste+explore vs reflect+observe
```

```
第 5 题：强度偏好
「你更想收到的推荐是——」
  A. 小的、随时可以做的事
  B. 大的、需要专门安排的事

→ 采集: intensity 偏好, micro vs event
```

**5 个问题，每个问题从原子组合中设计，答案直接映射为原子空间中的初始权重向量。** 不需要 AI，不需要后端，纯本地计算。

设计原则：
- 每个问题的两个选项都是"好的"——没有正确答案，用户不会感到被评判
- 问题本身就像卡片一样有趣——它不是测试，是体验的一部分
- 5 题已经足够在 6 维画像空间里锁定一个初始方向

#### 持续画像更新

每次交互更新画像权重，但权重不同：

| 行为 | 画像影响权重 | 理由 |
|------|------------|------|
| 右滑（想做） | 0.3 | 意愿信号，弱 |
| 左滑（不想） | -0.2 | 拒绝信号，比接受弱 |
| 长时间停留后左滑 | 0（不更新）| 犹豫 ≠ 不感兴趣，可能只是时机不对 |
| 打卡完成 | 1.0 | 真实行动，最强信号 |
| 打卡情绪正向 | 额外 +0.3 | 不只做了，还喜欢 |
| 打卡情绪"平淡" | 不额外加减 | 做了但没感觉，行动分照给，偏好分不加 |

半衰期 30 天——画像反映的是"你现在的生活状态"，而非"你曾经是谁"。

## 四、卡片系统重新设计

### 卡片的本质

卡片不是内容，是**一个行动建议**。它的价值不在于信息量，在于：
1. 让你觉得"这个我可以做"
2. 让你在做之前对体验有预期
3. 让你做完后觉得"值"

### 卡片数据结构（精简版）

```typescript
interface Card {
  id: string

  // 用户可见
  title: string          // 「去一家从没去过的餐厅」
  subtitle: string       // 一句话补充
  preview: string        // 预演文字（核心差异化）
  previewSummary: string // 预演摘要（一行）
  coverColor: string     // 主色调（Phase 0 用颜色代替图片）

  // 原子标签
  actions: ActionAtom[]  // 行为原子，1-2 个
  context: {
    social: SocialType
    space: SpaceType
    time: TimeType
    cost: CostType
    planning: PlanningType
    intensity: IntensityType
  }
  impact: Partial<Record<ProfileDimension, number>>  // 画像影响权重

  // 时效
  seasonal?: Season[]    // 适用季节，空=全年
  timeOfDay?: TimeOfDay[] // 适用时段，空=全天
}
```

### 卡片分层（简化为 3 层）

旧版有 5 层（微粒/日常/事件/大爆炸/里程碑），过多。简化为 3 层：

| 层 | 名称 | 时间 | 强度 | 占比 |
|----|------|------|------|------|
| 微 | 随手可做 | ≤30 分钟 | 低 | 40% |
| 中 | 需要安排 | 1-4 小时 | 中 | 40% |
| 大 | 值得期待 | 半天以上 | 高 | 20% |

**为什么砍掉大爆炸和里程碑？** 大爆炸卡（含子卡的复合体验）和里程碑卡（高考/考研）虽然设计有趣，但实现复杂度高，且偏离核心闭环。一个独立开发者应该先把"单卡闭环"做到极致，再考虑复合体验。后续如果需要，可以加回来。

### 预演文字：核心差异化

预演是这个产品最有价值的部分。它不是"介绍"，是**提前让你体验一遍**。

好的预演三要素：
1. **具体的时间锚点**：「头十分钟」「第三首歌的时候」「大约 30 分钟后」
2. **包含不确定性**：「可能会很好，也可能就那样」
3. **一个反直觉的洞察**：做之前想不到、但听了觉得"有道理"的东西

预演的 ROI 极高：纯文字，不需要技术实现，但直接决定用户是否行动。Phase 0 应该把大部分精力花在写好预演上。

## 五、核心交互流

### 主循环（极简）

```
打开 App
    │
    ▼
卡片流（一张一张看）
    │
    ├── 右滑 → 感兴趣 → 进入「想做」列表
    ├── 左滑 → 跳过（可选：不感兴趣 / 现在不合适）
    └── 点击 → 展开预演 → 看完后决定右滑或左滑

「想做」列表里的卡片
    │
    ├── 去做了 → 打卡（情绪 + 可选一句话）→ 庆祝
    ├── 没去做 → 7 天后开始视觉褪色
    └── 不想做了 → 移除

打卡完成
    │
    ├── 画像更新（用户看到维度变化）
    ├── 生成碎片（这次体验的凝固物）
    └── 推荐校准（下次推得更准）
```

### Phase 0 只需要 3 个屏幕

1. **发现页**：卡片流 + 滑动
2. **打卡页**：选情绪 + 可选备注
3. **庆祝页**：维度变化动画 + 回到发现

不做：个人主页、设置、队列管理、地图、碎片收藏。Phase 0 验证的是"卡片好不好 + 滑动爽不爽 + 打卡有没有成就感"。

## 六、推荐引擎

### Phase 0：规则引擎（纯本地）

```
全量卡片
  │
  ▼
① 状态过滤 → 去掉已完成/已拒绝/最近展示过的
  │
  ▼
② 时间过滤 → 晚 10 点不推"出门探索"；工作日不推"半天活动"
  │
  ▼
③ 偏好排序 → 原子权重点积：卡片原子 · 用户画像 = 匹配分
  │
  ▼
④ 强度渐进 → 打卡 < 3 次只推低强度；3-8 次引入中；8+ 引入高
  │
  ▼
⑤ 多样性保障 → 同一行为原子最多连续 2 张；至少 1 张 solo + 1 张 social
  │
  ▼
⑥ 野卡注入 → 每 5 张插 1 张偏好之外的，防信息茧房
  │
  ▼
最终卡片栈
```

这个引擎不需要后端，不需要 AI，纯前端几百行代码搞定。但用户感知是"它知道推什么给我"。

### 后续阶段：按需接入

| 能力 | 触发条件 | 接入方式 |
|------|---------|---------|
| AI 生成卡片 | 卡片池 < 200 张时需要扩充 | Claude API batch 生成 → 人工审核 |
| AI 预演生成 | 新卡片需要预演文字 | Claude API，用现有预演做 few-shot |
| 协同过滤 | 用户量 > 500 | Supabase 后端，简单的相似用户推荐 |
| 画像 Agent | 用户数据密度足够 | 基于画像数据的对话式 Agent |

**关键原则：不提前接入。** 每个能力在真正需要时才加入，不因为"技术上可以做"就提前做。一个独立开发者的时间是最稀缺的资源。

## 七、阶段规划（独立开发者视角）

### Phase 0：H5 Demo（1-2 周）

**目标**：发给 10 个朋友，验证两件事——
1. 卡片内容是否让人觉得"想做"
2. 滑动 → 打卡的闭环是否有成就感

**做**：
- 卡片滑动交互
- 30-50 张精品卡片（含预演文字）
- Taste Seeding 5 题快速画像
- 本地推荐引擎
- 打卡 + 情绪 + 维度变化动画
- Vercel 部署，分享链接

**不做**：账号、后端、地图、碎片、社交、队列管理

**技术**：React + Vite + Framer Motion + Zustand（延续现有原型）

**核心产出**：打卡率（右滑后真的去做了的比例）。如果 > 15%，核心假设成立。

### Phase 1：iOS App（4-6 周）

**目标**：100 人 TestFlight 内测，数据上云，开始积累真实画像数据

**新增**：
- 切到 Expo（React Native），复用业务逻辑层
- Supabase（Auth + DB），数据不再丢失
- 「想做」队列管理 + 卡片褪色
- 碎片系统（打卡后的永久记录）
- 维度画像可视化（简单的增量展示）
- 基础埋点（打卡率、留存、卡片级数据）

**视情况引入**：
- 推送通知（第一次打卡后申请权限）
- AI 批量生成新卡片（如果卡片池不够用）

### Phase 2：能力跃迁（视 Phase 1 数据）

**触发条件**：Phase 1 跑通，D7 留存 > 30%，有了几百个用户

**新增**：
- AI 预演文字（从用户打卡数据合成）
- 城市地图点亮
- 弱社交（同一张卡的匿名碎片互看）
- 协同过滤推荐
- 画像洞察（「你 60% 的体验都是独处的」）

### Phase 3：Agent 阶段（远期）

**触发条件**：画像数据密度足够，用户量达到一定规模

- 人格 Agent：基于画像数据的对话式助手，"真正懂你"
- 人格协议：开放规范，画像数据可迁移
- 智能回调：跨卡片语义关联推荐

## 八、独立开发者的资源分配建议

### 时间分配（Phase 0）

| 占比 | 投入在哪 |
|------|---------|
| 40% | **卡片内容 + 预演文字** — 这是产品灵魂，不能靠写代码解决 |
| 30% | **交互体验打磨** — 滑动手感、打卡流程、庆祝动画 |
| 20% | **推荐逻辑** — 让用户感觉"它懂我" |
| 10% | **其他**（部署、调试等） |

### 资金投入预估

| 阶段 | 月均成本 |
|------|---------|
| Phase 0 | ¥0（Vercel 免费） |
| Phase 1 | ~¥60/月（Apple Developer ¥99/年） |
| Phase 1+ | ~¥200/月（Supabase 超免费额度后 $25/月） |
| Phase 2 | ~¥500-1000/月（Claude API batch 生成） |

### 最大风险

**不是技术，是内容。** 卡片好不好、预演写得对不对、推荐时机准不准——这些决定产品成败，但无法用代码加速。

建议：Phase 0 阶段自己先做 10 张卡片的体验者——真的去做那些事。你自己的体验是最好的预演素材。

| 概念 | 来源 | 为什么保留 |
|------|------|-----------|
| 原子体系 | 卡片系统设计 | 整个产品的参数化基础 |

| 卡片褪色 | 产品全貌 v0.2 | "弱跟踪"的最优雅表达 |
| 逆向 Onboarding | 设计反思清单 | 用户先体验再被理解，而非先填问卷 |

## 十、这个产品最终在赌什么

赌一件事：**大多数人不是不想让生活更丰富，是缺少一个"推一把"。**

如果这个假设成立——人们看到一张写着"今晚上床不带手机"的卡片，会真的去做——那后面的一切（画像、Agent、人格协议）都顺理成章。

如果这个假设不成立——人们只是滑动收藏但从不行动——那这个产品就是又一个 Pinterest。

Phase 0 验证的就是这一件事。
