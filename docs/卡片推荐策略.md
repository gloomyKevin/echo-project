# 卡片推荐策略

> 本文档聚焦推荐逻辑的演进规划，不涉及卡片数据结构或交互设计（见卡片系统设计.md）。
> 核心约束：**本地阶段将持续相当长时间**，这不是过渡，是一个需要认真对待的完整产品形态。

---

## 两个阶段，一条分水岭

```
本地阶段 ──────────────────────┬────────────────── 服务端阶段
纯前端 + 卡片 JSON 增量更新     │    接入后端 + AI + 用户云数据
                               │
                         这是分水岭
                    不是优化，是能力跃迁
```

分水岭前后的根本差异：

| 能力 | 本地阶段 | 服务端阶段 |
|------|---------|----------|
| 用户画像 | 设备内行为推断 | 跨设备、跨时间、跨用户 |
| 推荐算法 | 规则 + 本地权重计算 | 协同过滤 + 个性化模型 |
| 内容更新 | 定期推送卡片 JSON | 实时生成、动态调整 |
| 预演内容 | 编辑先验 | 用户数据 + AI 合成 |

## 本地阶段：用规则逼近智能

### 可用信号（无需任何授权或服务端）

| 信号 | 来源 | 用途 |
|------|------|------|
| 当前时间 + 星期 | 系统 | 过滤可执行性：工作日晚上不推事件卡 |
| 历史右滑记录 | localStorage | domain 频次统计 → 偏好权重 |
| 打卡历史 + 情绪 | localStorage | 最高质量信号，权重 >> 右滑 |
| 卡片停留时长 | 前端埋点 | 区分「有兴趣但犹豫」vs「快速跳过」|
| 位置粗分类 | GPS（可选） | 在家/通勤/外出 → 可执行性筛选 |

### 推荐逻辑层次（本地，按优先级）

```
1. 硬过滤（可执行性）
   当前时间 + 地点 → 排除此刻做不到的卡片

2. 软排序（偏好匹配）
   右滑过的 domain/atoms → 对相似卡片加权
   打卡过的维度 → 适度注入对立维度（防信息茧房）

3. 多样性注入
   每 N 张插入一张与当前偏好不同的「野卡」
   保持发现感，防止推荐窄化

4. 新鲜度控制
   已右滑/已打卡的卡片降权或隐藏
   卡片 JSON 增量更新带来的新卡优先展示
```

### 软聚类（本地可实现）

不需要模型，就是权重比较：

```typescript
// 根据右滑 + 打卡历史，推断用户偏向
const profile = {
  indoor: swipes.filter(s => s.indoor).length,
  outdoor: swipes.filter(s => s.outdoor).length,
  solo: swipes.filter(s => s.solo).length,
  social: swipes.filter(s => s.social).length,
  reflective: domainCount('reflective'),
  exploratory: domainCount('exploration'),
}
// 软聚类：哪边更重，就归哪类
// 聚类结果影响野卡来源和排序权重
```

5-8 次有效滑动后即可生效，不需要等 20-30 次。

### 卡片 JSON 增量更新作为运营杠杆

本地阶段的唯一「后端动作」，但影响力不小：

- 季节/节气内容定期推送
- 根据全局打卡数据（汇总后人工分析）补充高置信卡片
- 修正预演文字（发现某张卡的预演和用户反馈不符时）
- **更新频率建议**：每 2-4 周一批，每批 5-10 张，保持新鲜感但不稀释质量

## 服务端阶段：能力跃迁后的扩展

### 新增能力

**协同过滤**：「和你行为相似的人还做过这些」——本地阶段完全没有这个能力，是服务端带来的最大增量。

**跨时间画像**：用户换设备不丢数据；长期行为模式（比如每年春天偏好的变化）才能被识别。

**预演动态化**：
- N > 30 完成：「已有 N 人做过，70% 选了「很满足」」
- N > 100：情绪分布可视化
- AI 合成阶段：从用户打卡文字归纳体验模式

**实时可执行性**：结合用户日历、天气 API、位置历史，做精细的时机推荐（「现在正好适合做这件事」）。

### 分水岭迁移原则

- Service Interface 层从 Day 1 就设计好，`LocalCardService` → `SupabaseCardService` 一行替换
- 本地阶段积累的行为数据在迁移时上传，不丢失冷启动信号
- 用户感知应该是「越来越准」的连续体验，而不是「切换了一个新系统」

---

## 始终不变的原则

无论本地还是服务端，推荐策略都要遵守：

- **打卡权重远大于右滑**：5-10 倍，防刷分，保证信号质量
- **30 天半衰期**：长期不活跃，偏好权重向均值漂移，分数不永久累积
- **野卡注入**：始终保留一定比例的意外推荐，产品的核心价值是「发现」而非「确认」
- **可执行性优先**：推一张此刻做不到的卡片，比推一张不够精准的卡片伤害更大

## 聚类设计反思

### 当前方案与已知缺陷

当前使用三类信号：`soloVsSocial`、`indoorVsOutdoor`、`domainWeights`。

**已知问题：**

1. **早期信号方差过大**：1-2 次滑动就会把某个 domain 的权重推到 0 或 1，导致推荐窄化。已用 Laplace 平滑修复（α=0.5，先验=中性，数据累积后逐步收敛）。

2. **solo/social 轴语义模糊**：很多卡同时标 `solo: true, social: true`（例如「发消息」可以独处发），这类卡在两组都被计入，对轴的方向没有区分贡献。轴的实际含义更接近「是否需要他人在场」，而非真正的性格内外向。

3. **缺少新奇偏好轴**：exploration/reflective 的区分（Big Five 开放性的代理变量）在当前方案里混在 domainWeights 里，没有单独建模。高开放性用户倾向探索类，低开放性偏内省类——这是比 indoor/outdoor 更有预测力的轴。

4. **强度偏好没有建模**：有些用户喜欢高挑战感（高强度右滑率高），有些喜欢低门槛（全部低强度右滑）。目前强度只做了进阶解锁，没有做偏好匹配。

5. **右滑 vs 左滑的不对称性被忽略**：强力左滑一个 social 卡的信号强度，理论上应该 > 轻度右滑一个 social 卡。目前都当等价对待。

### 业内参考方案

| 方案 | 核心逻辑 | 适用阶段 |
|------|---------|---------|
| Spotify 情绪-能量矩阵 | Russell 环形情绪模型：valence × arousal | 有大量内容元数据时 |
| Netflix 品味聚类 | 协同过滤 → 用户聚类 → 内容映射 | 需要用户规模 |
| Duolingo 可重复学习 | 记忆衰减曲线（间隔重复）| 学习类产品 |
| **PERMA 模型匹配** | 积极情绪/投入/关系/意义/成就 → 直接对应卡片 domain | **适合本产品** |

PERMA 模型是目前最值得借鉴的框架：六个维度（health/finance/career/social/environment/possibility）可以直接对应 Seligman 的五个 well-being 要素，而卡片的 domain 标签则对应实现这些要素的行为路径。

### Phase 0 可实施的改进方向

| 改进 | 优先级 | 备注 |
|------|--------|------|
| ✅ Laplace 平滑 | 已实施 | 解决早期信号抖动 |
| ✅ 右滑轻微影响维度分 | 已实施 | 体现意向信号 |
| 提取 noveltyAppetite 轴 | 中 | `exploration+creative` vs `reflective` 右滑率之比 |
| 时间段感知 | 低 | 早晨推晨间卡，晚间推内省卡 |
| 强度偏好建模 | 低 | 当前 checkin 计数进阶机制足够，过早加复杂度得不偿失 |

### 维度覆盖现状（v2 卡片池 59 张）

| 维度 | 有影响的卡片数 | 评估 |
|------|------------|------|
| 健康 | ~30 | 过多，许多 reflective 卡都附带健康影响 |
| 社交 | ~18 | 合理 |
| 环境 | ~15 | 合理 |
| 可能性 | ~22 | 合理，但定义仍偏模糊 |
| 成长/职业 | ~8 | c046-c049 补充后基本够用 |
| 财务 | ~5 | c050-c052 补充，Phase 0 上限合理——财务行为变化不是本产品核心 |

**财务维度的定位说明**：财务维度在 Phase 0 不应追求高覆盖。这个维度的作用是「消费意识觉醒」而非「理财建议」，3-5 张卡触发反思足够，不应成为产品的主旋律。
