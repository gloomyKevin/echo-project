# 技术底座与演进路径

> 解决两个问题：早期没有 AI 和后端资源时如何让产品跑通；真正具备能力后如何平滑迁移，而不是推倒重来。

---

## 一、底座的建设原则

早期最贵的错误不是功能做少了，而是**用错了架构，导致后期每次升级都要大规模重写**。

底座需要满足三个条件：

1. **接口稳定**：AI、后端、本地数据可以互换，但调用方的代码不动
2. **状态集中**：用户的所有状态（卡片、画像、等待盒子）都在一个地方，不散落在组件里
3. **渐进可替换**：每个「假」的实现，都有一个对应的「真」实现可以无缝替换进去

---

## 二、技术选型建议

### 端选择

移动端优先，但早期用 **Expo（React Native）** 做，理由：

- 一套代码跑 iOS + Android + Web（Web 用于早期演示和分享）
- Expo Go 可以扫码直接在手机上看效果，不需要打包
- 后期可以 eject 到原生，不是死路
- 卡片滑动、地图、相机、haptic feedback 都有成熟的 Expo 库

### 状态管理

**Zustand**，理由：

- 极简，一个 store 文件几十行
- 不依赖 React Context，性能好
- 持久化只需要加一个 middleware（`zustand/middleware` 的 `persist`），底层可以是 AsyncStorage 或 localStorage，一行切换

### 数据持久化抽象

做一个统一的 `StorageAdapter`，背后可以是：

- `Phase 0`：AsyncStorage（本地）
- `Phase 1`：Supabase（云端，自带 auth）
- 调用方不感知差别

## 三、核心状态机

产品的「聪明感」全部来自状态的正确流转。这套状态机是底座最核心的部分，Phase 0 就要建好，后续只是替换数据来源。

### 3.1 卡片状态机

每张卡片对每个用户有独立的状态：

```
UNSEEN
  │
  ▼
SHOWN
  ├── 左滑 ──────────→ REJECTED
  │                      └── (30天后可重新变为 UNSEEN，软重置)
  ├── 上滑 ──────────→ DONE_BEFORE（历史记录，不再推荐）
  └── 右滑 ──────────→ QUEUED
                          │
                          ├── 打卡完成 ──→ COMPLETED（永久档案）
                          ├── 主动放弃 ──→ SHELVED（进等待盒子收藏）
                          └── 超时未动 ──→ FADING（7天）→ ARCHIVED_WISH
```

状态枚举（TypeScript）：

```typescript
type CardStatus =
  | 'unseen'
  | 'shown'
  | 'rejected'
  | 'done_before'
  | 'queued'
  | 'fading'          // 右滑后超过 7 天未完成
  | 'archived_wish'   // 褪色后静默归档
  | 'shelved'         // 用户主动放入等待盒子
  | 'completed'       // 已打卡，永久

type WaitingReason =
  | 'season'          // 等季节
  | 'condition'       // 等条件（预算/地点）
  | 'mood'            // 等心情
  | 'saved'           // 主动收藏
```

### 3.2 用户画像状态

画像不是一次性填写的，是每次交互自然累积的：

```typescript
interface UserProfile {
  // 六维分数（0-100，从 50 开始，随交互漂移）
  scores: {
    health: number
    finance: number
    career: number
    social: number
    environment: number
    possibility: number
  }

  // 原子偏好向量（每个原子维度的权重，从 0 开始累积）
  atomWeights: {
    solo: number          // 偏独处
    social: number        // 偏社交
    indoor: number
    outdoor: number
    lowCost: number
    highCost: number
    physical: number      // 偏身体活动
    reflective: number    // 偏内省
    creative: number
    exploratory: number
  }

  // 聚类归属（Phase 1 之后才有意义）
  clusterWeights: Record<ClusterType, number>

  // 交互历史统计
  totalSwipes: number
  totalCompleted: number
  lastActiveDate: string
}
```

画像更新规则（Phase 0 纯本地计算）：

```typescript
function updateProfile(profile: UserProfile, event: CardEvent): UserProfile {
  switch (event.type) {
    case 'RIGHT_SWIPE':
      // 轻微增加卡片对应原子的权重
      return addAtomWeights(profile, event.card.atoms, 0.3)

    case 'LEFT_SWIPE':
      // 轻微降低
      return addAtomWeights(profile, event.card.atoms, -0.2)

    case 'COMPLETED':
      // 完成权重最高，同时调整六维分数
      const withAtoms = addAtomWeights(profile, event.card.atoms, 1.0)
      return applyDimensionDelta(withAtoms, event.card.dimensionImpact)

    case 'CHECKIN_EMOTION':
      // 情绪贴纸反向校正维度估算的准确性
      return calibrateDimensions(profile, event.emotion, event.card)
  }
}
```

### 3.3 每日卡片栈生成

Phase 0 完全本地，不需要服务器：

```typescript
function generateDailyStack(
  allCards: Card[],           // 静态 JSON
  userProfile: UserProfile,   // 本地状态
  cardStatuses: CardStatusMap, // 本地状态
  context: DailyContext       // 时间/位置
): DailyStack {

  const pipeline = allCards
    // ① 硬红线过滤（本地规则）
    .filter(card => !isHardBlocked(card))
    // ② 用户红线过滤（本地设置）
    .filter(card => !isUserBlocked(card, userProfile))
    // ③ 季节/时效过滤（本地时间判断）
    .filter(card => isSeasonallyValid(card, context.date))
    // ④ 可执行度过滤（本地上下文判断）
    .filter(card => getExecutability(card, context) !== 'gray')
    // ⑤ 去重（排除已展示/已完成）
    .filter(card => !isRecentlyShown(card, cardStatuses))
    // ⑥ 评分排序（本地原子权重计算）
    .map(card => ({ card, score: scoreCard(card, userProfile) }))
    .sort((a, b) => b.score - a.score)

  // ⑦ 多样性注入
  const diverse = ensureDiversity(pipeline.map(x => x.card))
  // ⑧ 野卡注入（随机从偏好以外选）
  const withWild = injectWildCards(diverse, allCards, userProfile)
  // ⑨ 精选一张
  const featured = pickFeatured(withWild, userProfile, context)

  return { cards: withWild, featured }
}
```

`scoreCard` 是核心评分函数，Phase 0 是原子权重的加权点积，后期可以换成服务端推荐分数，接口不变。

## 四、AI 依赖功能的本地替代方案

| 功能 | 真实需要 | Phase 0 替代 | 迁移方式 |
|------|----------|-------------|---------|
| 卡片推荐 | 服务端 ML 推荐 | 本地原子权重评分 | 替换 `scoreCard()` 的实现 |
| 选择预演文字 | AI 实时生成 | 静态 JSON 字段（人工编写） | 替换 `getPreview(cardId)` 的数据来源 |
| 维度影响分 | 模型推断 | 卡片 JSON 里的预设区间 | 同上，字段结构不变 |
| 用户画像推断 | 服务端模型 | 本地原子权重累积 | 替换 `updateProfile()` 的实现 |
| 智能回调 | 跨卡片语义关联 | 基于原子标签的硬规则 | 替换 `getRelatedCards()` 的实现 |
| 卡片生成 | AI 从原子组合生成 | 纯静态卡片库 | 增加 AI 生成通道，老卡片保留 |

**设计原则：每个依赖 AI 的功能，都通过一个函数接口暴露。Phase 0 是这个函数的「笨实现」，后期替换函数内部，调用方无感知。**

## 五、服务接口层（ServiceLayer）

这是整个架构里最重要的一层设计。所有「可能变化的依赖」都抽象成接口：

```typescript
// 卡片服务 —— 今天的推荐
interface CardService {
  getDaily(context: UserContext): Promise<Card[]>
  getCard(id: string): Promise<Card>
  getPreview(cardId: string, context: UserContext): Promise<CardPreview>
  getWaiting(userId: string): Promise<WaitingCard[]>
}

// 画像服务 —— 用户是谁
interface ProfileService {
  get(userId: string): Promise<UserProfile>
  applyEvent(userId: string, event: ProfileEvent): Promise<UserProfile>
  getInsight(userId: string): Promise<ProfileInsight>  // 「你最近喜欢独处」
}

// 地图服务 —— 热力数据
interface MapService {
  getHeatData(region: BoundingBox): Promise<HeatPoint[]>
  addCompletion(point: CompletionPoint): Promise<void>
}
```

### Phase 0 实现（纯本地）

```typescript
class LocalCardService implements CardService {
  private cards: Card[] = require('./data/cards.json')

  async getDaily(context: UserContext): Promise<Card[]> {
    const profile = profileStore.getState().profile
    const statuses = cardStatusStore.getState().statuses
    return generateDailyStack(this.cards, profile, statuses, context)
  }

  async getPreview(cardId: string): Promise<CardPreview> {
    const card = this.cards.find(c => c.id === cardId)
    return card?.preview ?? defaultPreview  // 静态字段
  }
}
```

### Phase 2 实现（真实后端）

```typescript
class RemoteCardService implements CardService {
  async getDaily(context: UserContext): Promise<Card[]> {
    return api.post('/cards/daily', { context })  // 完全一样的接口
  }

  async getPreview(cardId: string, context: UserContext): Promise<CardPreview> {
    return api.get(`/cards/${cardId}/preview`, { context })  // AI 生成
  }
}
```

**切换方式**：一行代码，注入不同的实现。

```typescript
// Phase 0
const cardService: CardService = new LocalCardService()

// Phase 2（改这一行，其他地方不动）
const cardService: CardService = new RemoteCardService()
```

用依赖注入或工厂模式管理，可以用环境变量控制：

```typescript
const cardService: CardService = __DEV__ || !API_ENABLED
  ? new LocalCardService()
  : new RemoteCardService()
```

## 六、数据结构（从第一天就定好）

卡片的 JSON 结构需要从一开始就考虑好，因为它贯穿整个生命周期：

```typescript
interface Card {
  id: string                    // 永久不变的 UUID
  version: number               // 内容版本号，用于缓存刷新

  // 用户可见内容
  title: string                 // 「去一家从没去过的餐厅」
  tagline: string               // 一句话
  coverImage: string            // 图片 URL 或本地资源路径
  scaleType: ScaleType          // micro | daily | event | bigbang | milestone

  // 原子标签（推荐引擎用）
  atoms: {
    trigger: TriggerAtom[]      // 适合出现的情境
    action: ActionAtom          // 核心动作类型
    context: ContextAtom        // 独处/社交，室内/室外等
    domain: Domain              // food | art | nature | social | ...
  }

  // 执行信息
  requirements: {
    minMinutes: number
    maxMinutes: number
    budgetMin: number           // 0 = 免费
    budgetMax: number
    seasonValid: Season[]       // [] = 全年
    locationRequired: LocationType  // home | nearby | city | travel
    socialType: SocialType      // solo | pair | group | stranger
  }

  // 预演内容（Phase 0 静态，Phase 2 由 API 返回覆盖）
  preview: {
    process: string             // 过程预演文字
    outcome: string             // 结果预演
    dimensionImpact: {
      [key in Dimension]?: { min: number; max: number }
    }
    emotionDistribution?: {     // 有真实数据后才填
      [key in Emotion]?: number  // 0-1 的比例
    }
    dataSource: 'editorial' | 'aggregated' | 'ai'
    sampleSize?: number         // 有数据时填
  }

  // 元数据
  globalCompletionCount: number  // 全局完成次数（定期从后端同步，本地缓存）
  createdAt: string
  tags: string[]
}
```

这个结构在 Phase 0 就能完整使用（除了 `emotionDistribution` 和远端的 `globalCompletionCount`），后期字段逐步填充，不需要改结构。

## 七、演进路径总览

### Phase 0 — 纯 Demo（当前）

**目标**：验证核心交互，卡片滑动 → 打卡 → 六维变化可见。

技术特征：
- 全部数据在 `cards.json` 静态文件
- 状态存 AsyncStorage（本地）
- 无用户账号，数据不跨设备
- 无真实地图热力，用假数据
- 预演文字全部人工写

**可以验证的问题**：交互流畅度、卡片内容质量、完成率是否够高、预演是否被阅读。

---

### Phase 1 — 轻后端（第一批真实用户）

**目标**：数据上云，多设备同步，开始积累真实的完成数据。

新增组件：
- Supabase（Auth + Database + Storage）—— BaaS，前端直连，不需要自己写服务器
- 卡片库改为从 Supabase 表读（可以在后台更新卡片内容不需要发版）
- 用户状态同步到云端（换手机数据不丢）
- 地图开始展示真实的完成点（匿名聚合）

**迁移方式**：把 `LocalCardService` 换成 `SupabaseCardService`，把 `AsyncStorage` 换成 `SupabaseStorageAdapter`。接口层完全不动。

**无需后端服务器**：Supabase 提供的 row-level security 可以让前端直接安全读写数据库，这个阶段不需要任何自己的服务器。

---

### Phase 2 — AI Preview 接入（数据量够了以后）

**目标**：预演文字开始从真实数据生成，推荐开始有意义。

新增组件：
- 第一批 AI 能力：用 batch 方式预生成热门卡片的预演文字（不是实时的，是离线跑的）
- 服务端推荐接口（简单的协同过滤，不需要大模型）
- 情绪分布数据开始展示（已有足够打卡数据）

**迁移方式**：`getPreview()` 开始从数据库里读 AI 生成的内容（而不是 JSON 里的静态字段），接口不变。`scoreCard()` 换成调用服务端接口，接口不变。

---

### Phase 3 — 全 AI（有工程资源时）

**目标**：实时推荐、实时预演生成、人格推断模型。

新增组件：
- 实时推荐服务
- AI 从原子体系实时生成新卡片（经过质量过滤后进入卡片池）
- 人格推断模型（从行为序列推断 Big Five 相关维度）

**迁移方式**：在已有接口层下，替换具体实现。这一步可以灰度——对 10% 的用户启用实时推荐，对比留存率，再全量。

## 八、冷启动阶段的假 AI 感知

在没有真正 AI 的情况下，有几种方式让产品「感觉聪明」：

### 1. 时间感知推荐

完全靠本地时钟做简单规则，但用户感知是「它知道现在几点」：

```typescript
function getTimeContext(): TimeContext {
  const hour = new Date().getHours()
  const day = new Date().getDay()

  return {
    period: hour < 6 ? 'late_night'
          : hour < 12 ? 'morning'
          : hour < 18 ? 'afternoon'
          : hour < 22 ? 'evening'
          : 'night',
    isWeekend: day === 0 || day === 6,
    isHolidayEve: isBeforeHoliday()  // 查本地节假日表
  }
}
```

22:00 工作日打开 App，看到的都是微粒卡和在家就能做的事——用户会觉得「它懂我」，但这完全是本地规则。

### 2. 滑动速度判断

```typescript
function inferConfidence(swipeDuration: number): SwipeSignal {
  if (swipeDuration < 400) return 'strong_reject'    // 快速左滑
  if (swipeDuration < 800) return 'moderate'
  return 'hesitant'                                   // 慢慢左滑 = 可能只是时机不对
}
```

慢慢左滑的卡片，3 周后重新出现一次。用户会觉得「它好像记得我对这张卡片有点犹豫」。

### 3. 渐进透明（让用户看到系统在学习）

```typescript
function getProfileInsight(profile: UserProfile): string | null {
  if (profile.totalSwipes < 20) return null  // 还不够说

  const topAtom = getTopAtom(profile.atomWeights)
  const templates = {
    solo: '你最近对独自完成的体验更感兴趣',
    outdoor: '你倾向于出门的体验，而不是待在家里',
    reflective: '你选的事情里，有很多是向内看的',
  }
  return templates[topAtom] ?? null
}
```

20 次交互后，卡片栈里出现一张「你的探索风格」小片段——完全本地计算，但感觉是 AI 洞察。

## 九、关键的几个决定（现在就要做）

| 决定 | 选择 | 理由 |
|------|------|------|
| 端 | Expo (React Native) | 移动端感知，一套代码，Web 可演示 |
| 状态管理 | Zustand | 极简，persist middleware 一行切换本地/云 |
| 后端（Phase 1）| Supabase | BaaS，前端直连，无需服务器，有 Auth |
| 卡片数据格式 | 文档里定义的 JSON Schema | 从第一天就用，后期直接迁到数据库 |
| 接口抽象 | Service Interface 模式 | 是后期平滑迁移的唯一保障 |
| AI 接入节奏 | batch 离线优先，实时最后 | 最小成本，最可控质量 |
